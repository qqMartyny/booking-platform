# Booking Platform — Architecture Decisions

## Цели проекта

Разработка полноценной платформы бронирования жилья (аналог Booking.com/Airbnb) с акцентом на получение практического опыта в:

- Проектировании сложных доменных моделей
- Гексагональной архитектуре (Ports & Adapters)
- Классических паттернах проектирования (GoF)
- Паттернах микросервисов (Outbox, Saga, Strangler Fig, Circuit Breaker, Retry)
- Асинхронном взаимодействии через Apache Kafka
- Кэшировании через Redis
- Spring Security (JWT, роли, авторизация)
- Kubernetes и Docker
- Observability (Prometheus, Grafana, distributed tracing)
- Написании качественных тестов (unit, integration)

Проект используется как портфолио и демонстрация навыков enterprise Java-разработки.

---

## Участники системы

- **Guest** — ищет и бронирует жильё
- **Host** — размещает объявления, управляет бронированиями
- **Admin** — модерация, поддержка

## Архитектурная стратегия

### Фаза 1 — Modular Monolith

Единое Spring Boot приложение с чёткими модульными границами. Каждый модуль — отдельный пакет, общение только через интерфейсы (порты). Никакого прямого доступа к чужим пакетам.

**Почему не сразу микросервисы:**
- Границы доменов должны устояться на практике
- Монолит проще разрабатывать и отлаживать на старте
- Strangler Fig позволит вынести сервисы без переписывания логики

### Фаза 2 — Инфраструктура

- API Gateway (Spring Cloud Gateway) перед монолитом
- Kafka вместо Spring Events для общения между модулями
- Redis для кэширования и хранения токенов

### Фаза 3 — Strangler Fig к микросервисам

Постепенный вынос модулей в отдельные сервисы. Порядок:
1. Notification Service (максимально независим)
2. Auth Service
3. Property Service
4. Booking Service
5. User Service

## Модули

### shared-kernel
Общие типы для всех модулей. Не содержит бизнес-логики.
- `DateRange` — value object для диапазона дат с валидацией
- `Money` — value object для денежных сумм
- `DomainEvent` — базовый интерфейс для доменных событий
- `PageResult<T>` — обёртка для пагинированных результатов

### booking-module
Управление бронированиями. Самый богатый доменной логикой модуль.

### property-module
Объекты размещения (отели, апартаменты, комнаты). Управление доступностью и ценообразованием.

### user-module
Профили гостей и хозяев.

### notification-module
Отправка email/push уведомлений. Слушает события из других модулей.

### auth-module
Регистрация, логин, выдача JWT токенов.

## Архитектура внутри каждого модуля — Hexagonal

```
module/
  domain/
    model/        ← чистые доменные объекты, без Spring/JPA аннотаций
    event/        ← доменные события
    port/
      in/         ← use case интерфейсы (driving ports)
      out/        ← исходящие интерфейсы (driven ports)
    service/      ← бизнес-логика, реализует use case интерфейсы
  adapter/
    in/
      rest/       ← контроллеры + DTO
      kafka/      ← Kafka consumers (если нужно)
    out/
      persistence/  ← JPA entities, Spring Data repositories
      messaging/    ← Spring Events (фаза 1) → Kafka (фаза 2)
```

**Ключевые правила:**
- Доменный слой не зависит ни от чего внешнего (Spring, JPA, Kafka)
- Контроллеры работают с DTO, сервисы — с доменными объектами
- Маппинг Request → Domain происходит в контроллере
- Маппинг Domain → Response происходит в контроллере

## Взаимодействие модулей

**Синхронное (через порты):**
Booking вызывает Property для проверки доступности комнаты при создании брони.

```
BookingService → RoomAvailabilityPort → PropertyService
```

**Асинхронное (через события):**
Booking публикует событие, Notification подписывается.

```
Фаза 1: BookingService → SpringEventPublisher → NotificationHandler
Фаза 2: BookingService → KafkaPublisher → [booking-events topic] → NotificationService
```

**Правило проверки границ модулей:**
Если удалить пакет модуля A — модуль B должен компилироваться. Если нет — есть нарушение границы.

## Booking — доменная модель

### Статусная машина

Статусная машина описывает все возможные состояния объекта `Booking` и допустимые переходы между ними. Бронь всегда находится ровно в одном статусе и может переходить только по разрешённым путям — это гарантирует консистентность бизнес-логики.
```
                    ┌─────────┐
                    │ PENDING │
                    └────┬────┘
         ┌───────────────┼──────────────┐
         ▼               ▼              ▼
    ┌─────────┐     ┌──────────┐   ┌──────────┐
    │APPROVED │     │ REJECTED │   │CANCELLED │
    └────┬────┘     └──────────┘   └──────────┘
         │
         ├──── гость запросил изменение дат ────┐
         │                                      ▼
         │                             ┌─────────────────┐
         │                             │ CHANGE_REQUESTED│
         │                             └────────┬────────┘
         │               ┌─────────────────────┤
         │               ▼                     ▼
         │          ┌──────────┐    хозяин одобрил → APPROVED
         │          │CANCELLED │    (новые или старые даты)
         │          └──────────┘
         ▼
    ┌──────────┐
    │COMPLETED │
    └──────────┘
```

**Переходы:**
- `PENDING` → `APPROVED` — хозяин одобрил (или InstantBook = true)
- `PENDING` → `REJECTED` — хозяин отклонил
- `PENDING` → `CANCELLED` — хозяин или гость отменили до решения
- `APPROVED` → `CHANGE_REQUESTED` — гость запросил изменение дат
- `APPROVED` → `CANCELLED` — гость отменил подтверждённую бронь
- `APPROVED` → `COMPLETED` — гость выехал (scheduled job)
- `CHANGE_REQUESTED` → `APPROVED` — хозяин одобрил (новые или старые даты)
- `CHANGE_REQUESTED` → `CANCELLED` — отмена во время рассмотрения запроса
- `REJECTED`, `CANCELLED`, `COMPLETED` — конечные состояния, переходов нет

### Два режима подтверждения
Хозяин настраивает на уровне объекта размещения:
- `InstantBook = true` → бронь автоматически `APPROVED` при создании
- `InstantBook = false` → бронь уходит в `PENDING`, хозяин вручную approve/reject

### BookingChangeRequest
Запрос на изменение дат — отдельная сущность (не поле в Booking). Хранит историю всех запросов на изменение.

## Паттерны проектирования (GoF)

### Creational

| Паттерн | Где применяется |
|---------|----------------|
| **Factory Method** | `Booking.create(...)` — статический фабричный метод вместо конструктора. Гарантирует что новая бронь всегда в статусе `PENDING` и генерирует доменное событие |
| **Builder** | Построение сложных поисковых фильтров (`PropertySearchFilter`) и конфигурационных объектов |

### Structural

| Паттерн | Где применяется |
|---------|----------------|
| **Adapter** | Адаптеры в гексагональной архитектуре — `JpaBookingRepository` адаптирует Spring Data к доменному порту `BookingRepositoryPort` |
| **Decorator** | Добавление логирования или метрик поверх сервисов без изменения их кода |
| **Facade** | Use Case интерфейсы (`CreateBookingUseCase`) скрывают сложность внутренней логики сервиса от контроллера |

### Behavioral

| Паттерн | Где применяется |
|---------|----------------|
| **Observer** | Доменные события — `BookingApprovedEvent` публикуется сервисом, `NotificationHandler` подписывается. Модули не знают друг о друге |
| **Strategy** | Стратегии ценообразования в Property модуле — базовая цена, выходные, сезонность, скидки. Легко добавлять новые стратегии |
| **State** | Управление переходами статусов брони. Каждый статус знает какие переходы для него допустимы |
| **Template Method** | Базовый класс уведомлений определяет скелет алгоритма отправки, подклассы реализуют конкретный канал (email, push) |
| **Chain of Responsibility** | Цепочка валидаторов при создании брони — проверка дат, доступности комнаты, лимитов пользователя |

## Паттерны микросервисов

| Паттерн | Где | Зачем |
|---------|-----|-------|
| **Outbox** | Booking модуль | Гарантированная доставка событий в Kafka. Событие сохраняется в той же транзакции что и бронь |
| **Saga** | Booking + Payment | Координация многошаговых операций с компенсирующими транзакциями |
| **Strangler Fig** | Весь проект | Постепенный переход к микросервисам без остановки системы |
| **Circuit Breaker** | Межсервисные вызовы | Устойчивость к падению зависимостей через Resilience4j |
| **Retry** | Межсервисные вызовы | Повторные попытки при временных сбоях |
| **API Gateway** | Входная точка | Централизованная аутентификация, rate limiting, routing |

## Redis

Используется в двух местах:
- **Auth модуль** — хранение refresh токенов, blacklist отозванных JWT
- **Property модуль** — кэширование результатов проверки доступности комнат

**Почему не кэшировать брони:**
Бронь часто меняет статус → инвалидация кэша сложна. При approve нужна актуальная информация о конфликтах — устаревший кэш может привести к двойному бронированию.

---

## Observability

- **Distributed Tracing** — Micrometer Tracing + Zipkin (traceId через все сервисы)
- **Metrics** — Micrometer + Prometheus + Grafana
- **Centralized Logging** — структурированные логи с traceId
